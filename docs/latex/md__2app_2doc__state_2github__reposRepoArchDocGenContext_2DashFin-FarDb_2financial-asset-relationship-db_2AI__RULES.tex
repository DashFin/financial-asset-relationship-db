This document outlines the core technologies and best practices for developing and maintaining the Financial Asset Relationship Network application.\hypertarget{md__2app_2doc__state_2github__reposRepoArchDocGenContext_2DashFin-FarDb_2financial-asset-relationship-db_2AI__RULES_autotoc_md88}{}\doxysection{\texorpdfstring{Tech Stack Overview}{Tech Stack Overview}}\label{md__2app_2doc__state_2github__reposRepoArchDocGenContext_2DashFin-FarDb_2financial-asset-relationship-db_2AI__RULES_autotoc_md88}
The application suite is built primarily with Python, leveraging specialized libraries for data handling, visualization, and web interface creation.


\begin{DoxyItemize}
\item {\bfseries{Python\+:}} The foundational programming language for all application logic, data processing, and backend operations.
\item {\bfseries{Gradio\+:}} Employed for creating user-\/friendly web interfaces for machine learning models or data science demos, as seen in the "{}\+Financial Asset Relationship Network"{} for visualizing complex graph structures and exploring asset relationships.
\item {\bfseries{Pandas\+:}} The go-\/to library for all data manipulation, analysis, and structuring of tabular data (Data\+Frames). It\textquotesingle{}s essential for preparing data for display and calculations.
\item {\bfseries{Num\+Py\+:}} Utilized for high-\/performance numerical operations, array computations, and statistical functions, particularly in graph algorithms and data generation.
\item {\bfseries{Plotly (plotly.\+graph\+\_\+objects, plotly.\+express, plotly.\+subplots)\+:}} The exclusive library for generating all interactive charts, graphs, and visualizations across the Gradio application. This includes candlestick charts, bar charts, pie charts, gauges, and 3D network visualizations.
\item {\bfseries{yfinance\+:}} Dedicated to fetching real-\/time and historical financial market data, including stock prices, company information, and news.
\item {\bfseries{dataclasses \& enum\+:}} Python\textquotesingle{}s built-\/in modules for defining structured data models (e.\+g., {\ttfamily Asset}, {\ttfamily Equity}, {\ttfamily Bond}) and enumerations (e.\+g., {\ttfamily Asset\+Class}, {\ttfamily Regulatory\+Activity}), ensuring type safety and clear data definitions.
\item {\bfseries{datetime\+:}} Used for handling and formatting dates and times, crucial for financial data timelines and event management.
\item {\bfseries{json\+:}} Employed for serializing and deserializing data, particularly for displaying structured information within the Gradio interface.
\end{DoxyItemize}\hypertarget{md__2app_2doc__state_2github__reposRepoArchDocGenContext_2DashFin-FarDb_2financial-asset-relationship-db_2AI__RULES_autotoc_md89}{}\doxysection{\texorpdfstring{Library Usage Rules}{Library Usage Rules}}\label{md__2app_2doc__state_2github__reposRepoArchDocGenContext_2DashFin-FarDb_2financial-asset-relationship-db_2AI__RULES_autotoc_md89}
To maintain consistency, performance, and readability, adhere to the following guidelines for library usage\+:


\begin{DoxyItemize}
\item {\bfseries{Web Application Frameworks\+:}}
\begin{DoxyItemize}
\item Use {\bfseries{Gradio}} for interactive demonstrations of complex models, graph visualizations, or schema exploration where a tabbed interface and direct input/output components are beneficial.
\end{DoxyItemize}
\item {\bfseries{Data Manipulation\+:}}
\begin{DoxyItemize}
\item Always use {\bfseries{Pandas}} Data\+Frames for any tabular data processing, filtering, aggregation, and transformation.
\item For low-\/level numerical operations, array manipulations, or statistical functions, {\bfseries{Num\+Py}} should be used.
\end{DoxyItemize}
\item {\bfseries{Charting and Visualization\+:}}
\begin{DoxyItemize}
\item {\bfseries{Plotly}} is the sole library for all graphical representations. This ensures a consistent interactive experience and visual style. Avoid other charting libraries.
\end{DoxyItemize}
\item {\bfseries{Financial Data Acquisition\+:}}
\begin{DoxyItemize}
\item {\bfseries{yfinance}} is the standard for retrieving public financial market data. Do not implement custom scraping or use other external financial APIs unless explicitly required and approved.
\end{DoxyItemize}
\item {\bfseries{Data Modeling\+:}}
\begin{DoxyItemize}
\item Define all structured data objects using {\bfseries{dataclasses}} for clarity and maintainability.
\item Use {\bfseries{enum}} for any fixed sets of choices or categories (e.\+g., asset classes, event types).
\end{DoxyItemize}
\item {\bfseries{Date and Time Handling\+:}}
\begin{DoxyItemize}
\item Utilize the {\bfseries{datetime}} module for parsing, formatting, and performing operations on dates and times.
\end{DoxyItemize}
\item {\bfseries{Structured Output\+:}}
\begin{DoxyItemize}
\item When displaying complex data structures in the UI, especially in Gradio, use {\bfseries{json}} for clear, formatted output.
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{md__2app_2doc__state_2github__reposRepoArchDocGenContext_2DashFin-FarDb_2financial-asset-relationship-db_2AI__RULES_autotoc_md90}{}\doxysection{\texorpdfstring{Directory and File Naming Conventions}{Directory and File Naming Conventions}}\label{md__2app_2doc__state_2github__reposRepoArchDocGenContext_2DashFin-FarDb_2financial-asset-relationship-db_2AI__RULES_autotoc_md90}

\begin{DoxyItemize}
\item {\bfseries{Directory Names\+:}} All directory names MUST be all lower-\/case (e.\+g., {\ttfamily src/pages}, {\ttfamily src/components}, {\ttfamily src/utils}).
\item {\bfseries{File Names\+:}} File names may use mixed-\/case (e.\+g., {\ttfamily Asset\+Relationship\+Graph.\+py}).
\end{DoxyItemize}\hypertarget{md__2app_2doc__state_2github__reposRepoArchDocGenContext_2DashFin-FarDb_2financial-asset-relationship-db_2AI__RULES_autotoc_md91}{}\doxysection{\texorpdfstring{Coding Guidelines}{Coding Guidelines}}\label{md__2app_2doc__state_2github__reposRepoArchDocGenContext_2DashFin-FarDb_2financial-asset-relationship-db_2AI__RULES_autotoc_md91}

\begin{DoxyItemize}
\item {\bfseries{Modularity\+:}} Create small, focused files and components. Aim for components that are 100 lines of code or less. Refactor large files when necessary.
\item {\bfseries{Readability\+:}} Write clean, well-\/commented code. Use meaningful variable and function names.
\item {\bfseries{Consistency\+:}} Adhere to the existing coding style and conventions of the project.
\item {\bfseries{Completeness\+:}} All implemented features must be fully functional with complete code; avoid placeholders or partial implementations.
\item {\bfseries{Error Handling\+:}} Do not use {\ttfamily try/catch} blocks for general error handling unless specifically requested. Allow errors to propagate for easier debugging and system-\/level error management.
\item {\bfseries{Simplicity\+:}} Prioritize simple and elegant solutions over overly complex or over-\/engineered designs. Focus on the user\textquotesingle{}s request and make the minimum necessary changes.
\item {\bfseries{Responsiveness\+:}} Always generate responsive designs for web interfaces.
\item {\bfseries{User Feedback\+:}} Use toast components or similar UI elements to inform the user about important events or actions.
\end{DoxyItemize}
